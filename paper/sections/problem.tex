\section{Problem and Motivation}

FLP 论文研究的是分布式系统中的容错共识问题。所谓共识，是指多个相互通信的进程需要最终对同一个值达成一致。在论文的基本模型中，每个进程一开始持有一个二进制输入值，取值为 $0$ 或 $1$；协议的目标是让没有故障的进程最终决定同一个二进制输出值。

这里的“协议”指每个进程共同遵守的一套确定性本地规则：进程一开始如何初始化，收到某条消息后如何更新本地状态，接下来发送哪些消息，以及在什么条件下宣布自己决定 $0$ 或 $1$。在 FLP 的形式化模型中，进程被看作确定性自动机；给定进程当前状态和收到的消息，它的下一步行为是确定的。

一个共识协议至少需要满足三个核心要求。

\begin{itemize}
  \item \textbf{一致性（agreement）}：不能有两个非故障进程决定不同的值。例如，如果 $P_1$ 决定 $0$ 而 $P_2$ 决定 $1$，那么协议违反一致性。
  \item \textbf{非平凡性（nontriviality / validity）}：协议不能通过永远输出同一个常数来“作弊”。例如“所有进程无论输入是什么都决定 $0$”虽然一致且会终止，但它完全无视输入。FLP 使用的非平凡性要求是：$0$ 和 $1$ 都应当在某些可达配置下可能成为决定值。
  \item \textbf{终止性（termination）}：在满足故障假设的合法执行中，协议最终必须做出决定。实际系统中通常希望所有非故障进程最终都决定；FLP 为了证明更强的不可能性，只需要考虑一个更弱的要求，即最终至少有某个进程进入决定状态。
\end{itemize}

FLP 关注的系统是完全异步的。这意味着系统没有全局时钟，没有消息延迟上界，没有进程运行速度上界或下界，也没有可靠的故障检测器。如果一个进程很久没有回应，其他进程无法判断它已经崩溃，还是仅仅运行得很慢，或者只是它发出的消息在网络中延迟很久。

论文考虑的故障也不是最强的拜占庭故障。拜占庭进程可能撒谎、发送矛盾消息甚至恶意破坏协议；FLP 只考虑更温和的 crash failure，也就是某个进程静默停止。停止之后它不再执行计算，也不再发送消息。论文的惊人之处在于：即使消息系统可靠，消息不会丢失、损坏或重复；即使最多只有一个进程 crash；即使故障进程没有任何恶意行为，确定性异步共识仍然不可能保证终止。

可以用分布式数据库提交问题来理解这个模型。假设有三个数据管理进程 $P_1,P_2,P_3$ 共同处理一笔交易。每个进程一开始根据本地情况提出一个意见：$1$ 表示可以提交（commit），$0$ 表示应该放弃（abort）。系统需要所有正常进程最终做出同一个决定。否则，如果 $P_1$ 决定提交而 $P_2$ 决定放弃，数据库副本就会进入不一致状态。

一个朴素协议是：每个进程广播自己的初始值，等待收到所有进程的初始值后，再按固定规则决定；例如“只要看到一个 $0$ 就决定 $0$，否则决定 $1$”。如果没有进程 crash，这个协议可以成功。虽然消息可能延迟，但每个进程最终都会发送消息，所有消息最终都会到达，因此所有进程最终能看到同一组输入并做出同一决定。

但只要允许一个进程 crash，这个朴素协议就可能永远不决定。例如 $P_3$ 在发送自己的初始值之前崩溃，$P_1$ 和 $P_2$ 就会一直等待 $P_3$ 的消息。由于协议规定必须收齐所有人的值才能决定，系统无法终止。

也许可以尝试修改协议，让进程不再等待所有人。但这正是异步系统的困难所在：如果 $P_1$ 很久没有收到 $P_3$ 的消息，它无法区分 $P_3$ 已经 crash 和 $P_3$ 只是很慢。太早忽略 $P_3$ 可能破坏一致性或非平凡性；一直等待 $P_3$ 又可能破坏终止性。

因此，FLP 论文真正回答的问题是：

\begin{quote}
在一个完全异步、消息可靠、最多只有一个进程可能 crash 的分布式系统中，是否存在一个确定性共识协议，能够同时满足一致性、非平凡性和保证终止？
\end{quote}

FLP 的答案是否定的。它不是说系统一定会决定出互相矛盾的值，也不是说现实系统永远无法实现类似共识的服务；它证明的是，在完全异步模型中，如果协议坚持保持一致性和非平凡性，那么总存在某个合法执行使协议永远无法进入决定状态。简言之，FLP 的核心不是“有 crash 就会决定错”，而是“为了不决定错，可能永远决定不了”。
