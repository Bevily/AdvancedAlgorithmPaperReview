\section{Background and Related Work}

FLP 论文之前，agreement 已经是分布式计算和容错系统中的核心问题。一条重要脉络来自 Byzantine agreement。Pease、Shostak 和 Lamport 研究了存在故障处理器时如何达成一致 \cite{pease1980reaching}；Lamport、Shostak 和 Pease 随后用 Byzantine Generals problem 将这一问题表述成经典模型 \cite{lamport1982byzantine}。这些工作说明，一致性问题并不是 FLP 首次提出的；FLP 的独特之处在于，它把注意力集中到更弱的 crash failure 和完全异步模型上，证明即使故障模型显著弱于 Byzantine faults，确定性共识仍然面临根本困难。

另一条直接相关的脉络来自分布式数据库和事务提交。事务提交要求所有参与数据管理进程一致地决定 commit 或 abort，因此是 consensus 的实际来源之一。Skeen 的 decentralized termination protocol \cite{skeen1982decentralized} 以及 Skeen 和 Stonebraker 对 crash recovery 的形式化研究 \cite{skeen1983formal} 都反映了当时数据库系统对可靠提交和恢复机制的需求。FLP 论文选择 transaction commit 作为主要动机，正是因为该问题把一致性、故障和终止性之间的冲突暴露得非常清楚。

FLP 之后，相关研究大致沿着“改变模型假设”或“放宽终止要求”的方向发展。随机化协议是一条重要路线。Ben-Or 提出了完全异步 agreement 的随机化协议 \cite{benor1983advantage}，其关键思想是用概率性选择绕开确定性协议的限制；Bracha 和 Toueg 也研究了 resilient consensus protocols \cite{bracha1983resilient}。这些工作并不否定 FLP，而是通过改变终止保证的形式，在概率意义上重新获得可行性。

另一条路线是弱化完全异步模型。Dwork、Lynch 和 Stockmeyer 提出了 partial synchrony 模型 \cite{dwork1988consensus}，把系统假设放在完全同步和完全异步之间。这个方向的重要性在于，它解释了许多现实系统为什么可以使用 timeout、leader 和重试机制：这些机制通常隐含了某种最终稳定的时间假设，而不是纯粹的完全异步模型。

故障检测器理论进一步刻画了绕开 FLP 需要什么额外信息。Chandra 和 Toueg 引入 unreliable failure detectors，并研究它们如何帮助异步 crash failure 系统解决 consensus \cite{chandra1996unreliable}；Chandra、Hadzilacos 和 Toueg 随后给出 solving consensus 所需的 weakest failure detector \cite{chandra1996weakest}。这类工作把 FLP 中“无法区分慢进程和崩溃进程”的直觉转化为更精确的问题：系统至少需要获得多少关于故障的信息，才能恢复 consensus 的可解性。

在系统实现层面，state machine replication 和 Paxos 将 consensus 发展成构建容错服务的基本工具。Schneider 的 state machine approach 将复制服务和操作顺序化联系起来 \cite{schneider1990state}；Lamport 的 Paxos 系列工作则给出了 crash fault 场景下影响深远的共识协议 \cite{lamport1998part,lamport2001paxos}。后来，PBFT 将 Byzantine fault-tolerant replication 推向实用系统 \cite{castro1999practical}，Raft 则以更易理解的方式组织 replicated log consensus \cite{ongaro2014search}。这些后续工作共同说明，FLP 并没有终结 consensus 研究，而是促使后续系统更清楚地说明自己依赖的时序、故障检测、多数派或随机化假设。
