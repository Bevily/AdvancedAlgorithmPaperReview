# Main Technique Notes: FLP 的核心技术

阅读对象：Fischer, Lynch, and Paterson, *Impossibility of Distributed Consensus with One Faulty Process*。

日期：2026-07-16

## 一句话概括

FLP 的主要技术不是设计一个新的共识算法，而是设计一种 **不可能性证明方法**：

> 只要系统是完全异步的，并且允许一个进程 crash，那么对于任何确定性共识协议，总能构造出一种合法执行，让系统一直停留在“还不能安全决定”的状态，因此永远不决定。

更通俗地说，FLP 做的事情像是在证明：

> 不管你写了什么共识程序，我都可以通过安排消息到达顺序和进程运行顺序，让你的程序一直处在“现在决定还太早”的尴尬位置。

这里的“我”不是恶意进程，而是异步系统本身允许的调度方式。消息不会丢，进程也不撒谎，只是消息可以被延迟很久，进程可以很久不运行。

## 先区分：FLP 不是说协议会决定错

这是理解这篇论文最重要的一点。

FLP 不是说：

```text
某个进程决定 0，另一个进程决定 1。
```

如果一个协议真的这样，它本来就不是正确的共识协议。

FLP 说的是：

```text
为了避免决定错，协议在某些合法执行中可能永远不敢决定。
```

所以 FLP 的核心矛盾是：

```text
想保证一致性 -> 不能太早决定
想保证终止性 -> 又必须最终决定
完全异步 + 一个 crash -> 没办法同时保证两者
```

## FLP 证明使用的几个“工具”

### 1. Configuration：系统快照

可以把 configuration 理解成系统在某一时刻的完整快照，包括：

- 每个进程当前内部状态是什么；
- 每个进程是否已经决定；
- 消息缓冲区里有哪些还没被送达的消息；
- 每个进程接下来收到消息后会怎么反应。

类比一下，configuration 就像游戏存档。只要存档一样，之后如果做同样的操作，就会走向同样的结果。

这很重要，因为 FLP 研究的是确定性协议。确定性意味着：

```text
同一个系统快照 + 同一个下一步事件 = 同一个新快照
```

协议本身是确定的，但异步调度是不确定的。也就是说，程序规则固定，但“下一步让谁运行、送哪条消息”可以有很多选择。

### 2. Event / Step：下一步发生什么

论文把一次基本动作看成一个 event，直观上就是：

```text
把某条消息交给某个进程，让这个进程执行一步。
```

这个进程收到消息后，可能会：

- 更新自己的本地状态；
- 给其他进程发送新消息；
- 进入决定状态；
- 什么重要事情也不做。

多个 event 连起来就是一个 schedule，也就是一段执行顺序。

### 3. Valency：当前局面还能走向哪些决定

这是 FLP 证明最关键的概念。

一个 configuration 可以按照未来可能决定的值分成三类：

#### 0-valent

从这个状态继续执行，不管后面怎么合法调度，系统最终只可能决定 `0`。

直观说：

```text
这个局面已经注定是 0 了。
```

#### 1-valent

从这个状态继续执行，不管后面怎么合法调度，系统最终只可能决定 `1`。

直观说：

```text
这个局面已经注定是 1 了。
```

#### bivalent

从这个状态继续执行，有些合法调度可能最终决定 `0`，另一些合法调度可能最终决定 `1`。

直观说：

```text
这个局面还没有被命运写死。
```

或者更通俗一点：

```text
系统还站在岔路口，往左可能决定 0，往右可能决定 1。
```

FLP 的证明目标就是构造一条无限执行，让系统永远停留在 bivalent 状态附近，永远不跨进已经决定的状态。

## 技术主线：三步证明

FLP 的证明可以理解成三步。

### 第一步：一定存在一个 bivalent 初始状态

因为协议要满足非平凡性，它不能永远固定输出 `0`，也不能永远固定输出 `1`。所以一定存在某些初始输入可以导致决定 `0`，某些初始输入可以导致决定 `1`。

现在想象所有进程的初始值从全 `0` 慢慢变成全 `1`：

```text
00000
10000
11000
11100
11110
11111
```

相邻两行只差一个进程的输入。

如果最开始某些配置会决定 `0`，最后某些配置会决定 `1`，那么在这条变化链上，一定存在一个边界：边界一侧“看起来会走向 0”，另一侧“看起来会走向 1”。

FLP 证明说，这个边界附近必然存在一个 bivalent 初始配置。

直观原因是：如果只改变一个进程的初始值，就能让系统从“必然决定 0”突然变成“必然决定 1”，那么当这个进程崩溃、不再透露自己的信息时，其他进程无法区分这两种情况。为了保持一致性，协议不能在这两种不可区分的执行中做出相互矛盾的决定。

所以，系统一开始就可能处在一种“还没注定结果”的状态。

### 第二步：从 bivalent 状态出发，总能再走到另一个 bivalent 状态

这是 FLP 最精妙的部分。

假设当前系统处于 bivalent 状态，也就是未来既可能决定 `0`，也可能决定 `1`。现在调度器准备安排某个事件 `e` 发生，例如：

```text
把消息 m 交给进程 p。
```

直觉上，协议可能希望这个事件成为“关键一步”：

```text
事件 e 发生前：系统还没确定
事件 e 发生后：系统就注定决定 0 或注定决定 1
```

FLP 要证明的是：调度器总能先绕一绕，安排一些别的步骤，再让 `e` 发生，使系统仍然保持 bivalent。

这句话比较抽象，可以用一个比喻理解：

```text
系统站在岔路口。
协议希望下一步把系统推向某一条确定道路。
FLP 证明调度器总能换一种合法走法，让系统走了一小段之后仍然站在新的岔路口。
```

为什么可以这样做？核心原因是完全异步系统中，不同进程的很多步骤可以交换顺序。

如果两个事件发生在不同进程上，例如一个是 `p` 运行一步，另一个是 `q` 运行一步，那么在许多情况下：

```text
先 p 后 q
```

和

```text
先 q 后 p
```

会到达可以相互比较的状态。论文用这个“可交换性”证明：如果某一步真的能把 bivalent 状态强行变成 0-valent 或 1-valent，那么会推出两个本应不同的决定状态其实可以被连接起来，最后违反一致性或违反 valency 的定义。

通俗说，异步系统给了调度器太多自由。协议想依靠某个“关键消息”或“关键进程步骤”把结果定下来，但调度器可以把其他不冲突的事件插到前面，让协议始终无法把决定安全地锁死。

### 第三步：不断重复第二步，构造一个永不决定的合法执行

有了前两步之后，FLP 的最终构造就很自然：

1. 从一个 bivalent 初始配置开始。
2. 取出当前可以发生的某个事件。
3. 根据第二步，安排一段合法调度，使系统到达另一个 bivalent 配置。
4. 重复这个过程。

这样就得到一条无限执行：

```text
bivalent -> bivalent -> bivalent -> bivalent -> ...
```

由于一旦某个进程决定了 `0` 或 `1`，系统就不可能仍然是 bivalent，所以这条执行中没有进程做出决定。

也就是说，协议没有决定错，但它永远没有决定。

这就违反了 termination。

## 为什么这条无限执行是合法的？

一个常见疑问是：

> 你一直拖延关键消息，这是不是作弊？这还算合法执行吗？

FLP 的构造需要保证它不是作弊。论文中的无限调度会让消息和进程按照某种公平方式不断获得机会。简单理解就是：

- 不是把所有消息都永远丢掉；
- 不是让所有进程都永远停住；
- 最多只相当于有一个进程可能一直不继续执行，也就是允许的 one faulty process；
- 其他消息和进程仍然可以继续被调度。

在完全异步模型中，消息延迟可以任意长，进程也可以任意慢。因此，只要构造没有超出“最多一个进程 crash / 不继续执行”的限制，它就是模型允许的执行。

这也是 FLP 很强的地方：它不是依赖不可靠通信，也不是依赖大量进程崩溃，而是在非常温和的故障条件下构造了一个合法但永不决定的执行。

## 技术直觉：为什么“慢”和“死”会造成不可能性？

可以用三进程例子理解：

```text
P1, P2, P3
```

如果 `P3` 很久没有回复，`P1` 和 `P2` 面临两种可能：

```text
情况 A：P3 已经 crash。
情况 B：P3 只是很慢，它之后还会带着重要信息回来。
```

在完全异步系统里，`P1` 和 `P2` 没有办法区分 A 和 B。

如果协议为了终止而忽略 `P3`：

```text
那在情况 B 中，可能过早决定，和 P3 以后带来的信息冲突。
```

如果协议为了安全而等待 `P3`：

```text
那在情况 A 中，可能永远等待。
```

FLP 的 valency 证明就是把这个直觉形式化。它不是只说某个朴素协议会卡住，而是说任何确定性协议都会遇到这种不可区分性。

## 主要技术的非凡之处

### 1. 它证明的是所有协议，不是某个协议

很多反例只能说明：

```text
这个算法不行。
```

FLP 证明的是：

```text
这一类模型下，所有确定性算法都不行。
```

这需要抽象掉具体算法细节，只抓住所有异步确定性协议共同具有的结构：configuration、event、schedule、decision state。

### 2. 它把“直觉困难”变成了严格证明

“慢进程和死进程不可区分”只是直觉。FLP 真正厉害的是把它转化成：

```text
bivalent 初始配置存在；
bivalent 状态可以被延续；
因此存在无限 bivalent 执行；
因此违反 termination。
```

这让不可能性不再只是经验判断，而是形式化定理。

### 3. 它没有依赖恶劣条件

FLP 不需要：

- 消息丢失；
- 消息损坏；
- Byzantine 恶意行为；
- 多个进程同时崩溃；
- 网络永久分区。

它只需要：

```text
完全异步 + 一个 crash 的可能性 + 确定性协议
```

这就是结果让人意外的地方。

## 和后续系统的关系

FLP 并不是说现实系统不能做共识。它说的是，在完全异步模型中不能无条件保证确定性终止。

后续系统通常通过改变某个条件来绕开 FLP：

- Paxos / Raft：保证安全性，但活性依赖 leader、timeout、多数派和最终稳定的网络条件；
- partial synchrony：假设系统最终会有某种时间上界；
- failure detectors：给系统一些关于谁可能崩溃的额外信息；
- randomized consensus：不再要求确定性终止，而是要求概率意义上的终止。

所以 FLP 对工程系统的启发是：

```text
不要只说“我们能达成共识”。
必须说清楚：在什么额外假设下能达成共识。
```

## 可以写进正文的表达草稿

FLP 的核心技术是 valency-based impossibility proof。论文首先将分布式系统状态抽象为 configuration，并将异步执行表示为一系列 process step / message delivery events。随后，论文用 bivalent 和 univalent 刻画一个 configuration 未来可能导向的决定值：如果从某个 configuration 出发既存在决定 0 的合法延伸，也存在决定 1 的合法延伸，则该 configuration 是 bivalent；如果所有合法延伸只能导向同一个决定值，则它是 univalent。

基于这一抽象，FLP 证明分为两个关键部分。第一，由非平凡性可推出存在 bivalent initial configuration。第二，对于任意 bivalent configuration 和任意下一步事件，调度器总能构造一段合法执行，使系统在执行该事件后仍然到达某个 bivalent configuration。这个结论依赖异步系统中事件顺序的可交换性，以及进程无法区分消息延迟和进程崩溃的事实。反复应用该结论，就可以构造一条无限合法执行，使系统始终保持 bivalent，因此没有任何进程进入决定状态。这条执行违反 termination，从而证明不存在能够容忍一个 crash failure 的确定性完全异步共识协议。

## 仍需核对的问题

- 正文写作时，需要区分论文中的 partial correctness、total correctness 和本 review 中使用的 agreement / nontriviality / termination 表述。
- “合法执行”的公平性条件需要在证明章节更严谨地表述，避免让读者误以为证明只是无限期丢弃消息。
- `bivalent` 和 `univalent` 的正式定义应与原论文保持一致：它们描述的是从某个 configuration 可达的 decision value 集合，而不是当前已经决定的值。
