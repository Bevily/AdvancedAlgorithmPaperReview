# Problem Understanding: FLP Consensus Impossibility

Target paper: *Impossibility of Distributed Consensus with One Faulty Process*.

## What Problem Does This Paper Address?

这篇论文研究的是分布式系统中的 **consensus problem**，即在一个由多个进程组成的系统中，所有没有故障的进程如何对同一个二进制值达成一致。

更具体地说，每个进程一开始有一个输入值，通常是 `0` 或 `1`。协议希望最终让进程做出决定。这里的 **协议** 指每个进程共同遵守的一套确定性规则：进程一开始做什么，收到某条消息后如何改变本地状态，接下来给谁发送什么消息，以及什么时候可以宣布自己决定 `0` 或 `1`。在 FLP 论文中，协议被形式化为一组确定性进程自动机：给定进程当前状态和收到的消息，它的下一步行为是确定的。

共识协议希望满足以下要求：

- Agreement: 不同的非故障进程不能决定不同的值。
- Nontriviality / Validity: 决定值不能是一个完全无视输入的固定常数；`0` 和 `1` 都应该在某些初始配置下可能成为决定值。
- Termination: 在允许的故障范围内，系统最终应该至少有某个非故障进程做出决定。实际应用中通常希望所有非故障进程最终决定，但 FLP 证明只需要使用这个更弱的终止要求。

论文关注的系统模型非常关键：

- 系统是完全异步的。
- 进程之间没有统一时钟。
- 进程运行速度没有上界或下界。
- 消息延迟可以任意长。
- 消息系统是可靠的：消息不会丢失、不会损坏、不会重复，最终会被投递给持续接收的非故障进程。
- 故障类型不是 Byzantine failure，而只是最简单的 crash failure，即某个进程停止运行。
- 最多只有一个进程会发生 crash。

因此，这篇论文真正要回答的问题是：

> 在一个完全异步、消息可靠、最多只有一个进程会崩溃停止的分布式系统中，是否存在一个确定性共识协议，能够保证所有允许的执行都最终达成一致？

FLP 的答案是否定的。

论文证明：即使消息完全可靠，即使只有一个进程可能崩溃，即使故障不是恶意 Byzantine 行为，任何确定性异步共识协议都存在某种合法执行，使系统永远无法做出决定。

## Explanation of Key Terms

### Consensus

Consensus 可以理解为“多个进程最终对同一个值达成一致”。例如有三个进程 `P1, P2, P3`，它们一开始各自有一个输入：

```text
P1 = 1
P2 = 1
P3 = 0
```

共识协议的目标不是让每个进程保留自己的输入，而是让所有没有故障的进程最终决定同一个输出值，比如都决定 `0`，或者都决定 `1`。不能出现 `P1` 决定 `0` 而 `P2` 决定 `1` 的情况。

### Agreement / 一致性

一致性指：不能有两个非故障进程决定不同的值。

违反一致性的例子：

```text
P1 决定 0
P2 决定 1
```

不违反一致性的例子：

```text
P1 决定 0
P2 决定 0
P3 没有决定
```

一致性关注的是已经做出的决定之间不能互相矛盾。

### Nontriviality / Validity / 非平凡性

非平凡性是为了排除一种“作弊协议”。例如协议规定：

```text
所有进程永远决定 0。
```

这个协议确实永远一致，也会终止，但它完全无视输入。如果所有进程一开始都输入 `1`，协议仍然决定 `0`，这样的共识没有实际意义。

所以 FLP 要求协议不是固定输出常数。用较直观的话说，决定结果必须和输入有某种真实关系。FLP 论文使用的形式较弱：`0` 和 `1` 都应该在某些可达配置下可能成为决定值。

### Termination / 必然终止

必然终止指：在满足故障假设的合法执行中，协议最终必须做出决定。实际系统中通常希望所有非故障进程都最终决定。FLP 论文为了证明不可能性，甚至使用了更弱的要求：最终至少有某个进程做出决定。

如果系统一直运行、消息也在被调度，但没有进程进入决定状态，那么就是没有终止。

### 永远不决定

“永远不决定”不是说系统完全停了，而是说进程可能仍在运行、仍在接收或等待消息，但没有进程宣布最终决定。例如：

```text
P1: 我还没有收到 P3 的消息，不能决定。
P2: 我还没有收到 P3 的消息，不能决定。
P3: 已经 crash，不会再发送消息。
```

如果 `P1` 和 `P2` 永远等下去，就出现了永远不决定。

### Crash Failure

Crash failure 指进程静默停止。它停止之后不再执行本地计算，也不再发送消息。它不是 Byzantine failure：不会撒谎，不会恶意发送互相矛盾的消息，也不会伪造信息。

FLP 的结果很强，因为它不需要考虑恶意故障；仅仅一个进程静默停止，就足以破坏确定性异步共识的保证终止性。

### Completely Asynchronous System / 完全异步系统

完全异步指系统没有时间保证：

```text
进程运行速度没有上界或下界；
消息延迟没有上界；
没有全局时钟；
没有可靠的超时机制；
没有完美的故障检测器。
```

因此，如果 `P1` 很久没有收到 `P3` 的消息，`P1` 无法判断：

```text
P3 已经 crash；
P3 没有 crash，只是运行很慢；
P3 发了消息，但消息在网络中延迟很久。
```

这就是“慢进程和死进程不可区分”的含义。

## Example: Distributed Commit

一个典型例子是分布式数据库的提交问题。假设有三个数据管理进程：

```text
P1, P2, P3
```

它们共同处理一笔交易，例如从账户 A 转账 100 元到账户 B。最后所有正常进程必须做同一个决定：

```text
1 = commit，提交交易
0 = abort，放弃交易
```

如果 `P1` 决定 commit，而 `P2` 决定 abort，数据库副本就会不一致。因此这类系统天然需要共识。

一个最朴素的协议是：

```text
协议 A：等待所有人协议

1. 每个进程广播自己的初始值。
2. 每个进程等待收到所有进程的初始值。
3. 收齐后按固定规则决定：
   - 如果所有值都是 1，就决定 1；
   - 只要有一个值是 0，就决定 0。
```

如果零个进程 crash，这个协议可以成功。因为每个进程最终都会发送消息，每条消息最终都会到达。于是所有进程最终看到同一组输入，并按同一规则决定同一个值。

但如果允许一个进程 crash，协议就可能失败。例如 `P3` 在发送自己的输入之前 crash：

```text
P1 等待 P3 的值；
P2 等待 P3 的值；
P3 已经停止。
```

由于协议规定必须等所有人的值，`P1` 和 `P2` 会永远等待，无法终止。

也许可以修改协议，不等所有人，只等一部分进程。但这又带来另一个风险：在完全异步系统中，某个没有回应的进程可能不是死了，而只是很慢；如果太早决定，可能破坏一致性或非平凡性。FLP 证明的厉害之处在于，它不是只说明这个朴素协议不行，而是证明任何确定性协议都无法完全避免这种困境。

## Zero Crash, One Crash, and Many Crashes

零个 crash 时，异步共识相对容易。进程可以等待所有人的消息，因为没有进程会永久消失，所有消息最终都会到达。

允许一个 crash 后，问题发生本质变化。协议不能简单等待所有人，因为某个进程可能已经永远停止；但协议也不能随便忽略没有回应的进程，因为在异步模型中它可能只是很慢。于是协议在“安全地保持一致”和“保证最终决定”之间出现冲突。

多个 crash 不需要额外证明就更困难：如果一个协议能容忍多个 crash，它当然也应该能容忍只有一个 crash 的情况。但 FLP 已经证明，容忍一个 crash 并保证确定性异步共识终止都不可能。因此 “one faulty process” 的意义在于：最轻微的非零故障已经足以导致不可能性。

## Important Correction

确定性协议不意味着系统只有一条确定的执行轨迹。即使每个进程的程序是确定的，异步系统中消息先后顺序、进程调度顺序仍然可以不同。因此同一个确定性协议可能有许多可能执行。

FLP 也不是说一个正确协议会让不同进程决定不同值。正确协议必须保持一致性，所以它不能允许 `P1` 决定 `0` 而 `P2` 决定 `1`。FLP 的核心结论是：如果协议坚持一致性和非平凡性，那么在允许一个 crash 的完全异步系统中，总存在某种合法执行使协议永远不决定。

换句话说：

> FLP 不是“有 crash 就会决定错”，而是“为了不决定错，可能永远决定不了”。

## Why This Is Subtle

直觉上，一个进程崩溃似乎不是特别严重。如果消息可靠，并且只有一个进程可能停止，那么我们可能希望其他进程可以继续通信，最终绕过故障进程完成决定。

但完全异步模型中有一个根本困难：一个进程无法区分“另一个进程已经崩溃”与“另一个进程只是非常慢”。

如果协议因为怀疑某进程崩溃而继续决定，那么在另一个执行中，这个进程可能其实只是延迟很久才运行，过早决定可能破坏一致性。反过来，如果协议一直等待以避免错误决定，那么调度器可以不断延迟关键消息或关键进程，让协议永远保持在尚未决定的状态。

FLP 论文抓住的正是这个矛盾。

## A More Review-Oriented Formulation

如果写进 review，可以把问题表述为：

> FLP 论文研究的是异步消息传递系统中的容错共识问题。给定若干确定性进程，每个进程初始持有一个二进制输入值，目标是在最多一个进程可能 crash 的情况下，使非故障进程最终对同一个二进制值做出决定。论文证明，在没有时钟、没有消息延迟上界、没有故障检测器的完全异步模型中，这样的协议不可能同时满足一致性、非平凡性和保证终止。换言之，哪怕通信可靠、故障也只是一个进程静默停止，确定性异步共识仍然存在合法执行会永不终止。

## What The Paper Is Not Saying

- 它不是说现实系统不能实现看起来像共识的服务。
- 它不是说同步系统中共识不可能。
- 它不是说随机化协议不可能。
- 它不是说加入超时、失败检测器、部分同步假设后仍然不可能。
- 它不是说协议一定会在所有执行中失败，而是说没有协议能保证在所有允许执行中终止。

## Key Phrase To Remember

FLP 证明的核心问题不是“进程会坏”，而是：

> 在完全异步系统中，慢进程和死进程不可区分。

这个不可区分性使得协议无法同时保证安全地一致决定和在所有合法执行中必然终止。
